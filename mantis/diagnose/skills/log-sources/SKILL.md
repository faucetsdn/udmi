# UDMI Diagnostic Log Sources Reference

This guide catalogs the different log files generated during a UDMI test iteration. It explains where each file resides, its purpose, and the key search strings used to trace behaviors.

---

## 1. Support Package Structure

A standard diagnostic run folder contains the following file hierarchy:

```
run_X/
├── sequencer.out            # Overview of sequencer results (JUnit format)
├── test_itemized.out        # Overview of itemized test results (if run)
├── pubber.log               # Global stdout/stderr from the device emulator
├── udmis.log                # Global stdout/stderr from the cloud service pod
└── out/
    └── devices/
        └── <device_id>/
            └── tests/
                └── <test_name>/
                    ├── sequence.log  # Console log of the specific test case
                    └── sequence.md   # Readable markdown trace of the test case
```

---

## 2. Log File Catalogs & Patterns

### 1. The Local Test Sequence Log (`sequence.log`)
*   **Path:** `out/devices/<device_id>/tests/<test_name>/sequence.log`
*   **Purpose:** Captures the exact console output generated by the Sequencer while executing this specific test.
*   **Key Patterns to Search For:**
    *   `Starting test <test_id>`: Marks the start of the test case execution.
    *   `Ending test <test_id>`: Marks clean test completion.
    *   `ERROR terminating test`: Indication of a critical crash or early abort.
    *   `configTransaction RC:xxxxxx`: Dispatches a configuration message with the specified Base Transaction ID.
    *   `configSync error / timeout`: Indicates that the device failed to synchronize its state configuration within the timebound.

### 2. The Local Test Markdown Trace (`sequence.md`)
*   **Path:** `out/devices/<device_id>/tests/<test_name>/sequence.md`
*   **Purpose:** An autogenerated, high-level markdown timeline of the test execution.
*   **Key Value:** It lists every sent configuration payload and received state payload side-by-side in JSON code blocks, making it ideal to visualize schema properties.

### 3. The Global Cloud Pod Log (`udmis.log`)
*   **Path:** `udmis.log`
*   **Purpose:** Logs cloud-side packet processing, route translations, and registry reflections.
*   **Key Patterns to Search For:**
    *   `ReflectProcessor Processing reflection state/validation <timestamp> RC:<base>.<suffix> for <device_id>`: Indicates UDMIS is processing a validation or state message from the reflection control plane.
    *   `ReflectProcessor Propagating message RC:<base>.<suffix>: ValidationState`: UDMIS is propagating the validated state package back to the Sequencer.
    *   `DynamicIotAccessProvider Resolved provider`: Traces which cloud provider UDMIS routed the message through. Look for transitions between `clearblade-iot-core`, `google-iot-core`, or `mock-iot-core`.

### 4. The Global Device Emulator Log (`pubber.log`)
*   **Path:** `pubber.log`
*   **Purpose:** Logs the internal state, MQTT handlers, and loop timers of the Pubber emulator.
*   **Key Patterns to Search For:**
    *   `Connection to broker established`: Device connected to the MQTT message broker.
    *   `Received config RC:<base>.<suffix>`: Pubber received a new configuration transaction.
    *   `Publishing state (version <num>)`: Pubber published its current state package as a response to a configuration update.
    *   `Device simulation error / exception`: Indicates emulator runtime errors.
    *   > [!NOTE]
        > If the test was executed against physical hardware, `pubber.log` will be empty or absent. The debugging agent must treat the device as a "black box" and rely entirely on `sequence.log` and `udmis.log` traces.

## 3. Summary Results Indexes

### 1. `sequencer.out`
*   **Format:** Line-based space-delimited log.
*   **Purpose:** Lists the test result for every execution run.
*   **Format Pattern:** `RESULT <pass|fail> <category> <test_name> <details>`
*   **Example:** `RESULT fail system.last_update valid_serial_no last_start not synced`

### 2. `test_itemized.out`
*   **Format:** Identical to `sequencer.out`.
*   **Purpose:** Contains results for itemized schema validation tests.
*   **Example:** `0100 RESULT pass pointset.extra_point extra_point_ignored`
